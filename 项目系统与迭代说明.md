# 《黄昏居酒屋》项目系统与迭代说明

更新时间：2026-04-18
文档定位：项目级系统核查、扩展边界与可迭代性说明

## 1. 这份文档解决什么问题

`剧情文件规范文档.md` 主要回答“剧情节点应该怎么写”。  
本文件回答的是另一类问题：

- 目前项目真实实现到了哪里。
- 哪些目录和字段已经被运行时真正消费。
- 哪些功能只是有资产/占位，但还没有正式接线。
- NPC 大模型个性化对话应该接在哪里才不破坏主线。
- 后续怎样改，才能保证项目越做越容易扩展，而不是越做越乱。

## 2. 当前项目的真实实现边界

以下结论基于当前仓库代码与资产结构核查得出。

### 2.1 已经落地的功能

#### A. 主流程与相位切换

- 已有开始页、主菜单、开场序章、营业内主界面。
- 游戏主流程已收敛到统一状态系统，正式根状态为：
  - `boot`
  - `startScreen`
  - `mainMenu`
  - `introSequence`
  - `dayLoop.intro`
  - `dayLoop.guest.story`
  - `dayLoop.guest.chat`
  - `dayLoop.guest.observation`
- `dayLoop.guest.mixing`
- `dayLoop.guest.result`
- `dayLoop.guest.reward`
- `dayLoop.guest.llmChatLobby`
- `dayLoop.guest.llmChatSession`
- `dayLoop.guest.reflection`
- `dayLoop.daySummary`
- `App.tsx` 已从总流程控制器降为“状态机宿主 + 视图壳层”。

#### B. 来客排班与剧情入口

- `src/assets/schedule.yaml` 已驱动按周/按日来客。
- `getGuestsForDay()` 与 `findScheduledVisit()` 已支持顺序扫描后续来客。
- 当前排班数据已至少覆盖第一周 7 天。

#### C. 剧情节点驱动

- 角色剧情已采用 YAML 节点驱动。
- 运行时当前会加载：
  - `character_meta.yaml`
  - `nodes_main.yaml`
  - `nodes_teaching.yaml`
  - `nodes_chat.yaml`
  - `nodes_hidden.yaml`
  - `observations.yaml`
  - `gallery.yaml`
- `StoryPhase` 已支持：
  - `script_flow`
  - `player_options`
  - `trigger_observation`
  - 进入调酒节点
  - 记录当前客人剧情 transcript

#### D. 观察系统

- 已有独立 `ObservationPhase`。
- 已支持从剧情节点触发观察。
- 已支持 `feature_groups` 分组控制可观察特征。
- 观察结果会写回：
  - 当前轮次发现线索
  - 角色观察记录

#### E. 调酒与结果判定

- 已有独立 `MixingPhase`。
- 已支持：
  - 教学调酒引导
  - 普通调酒提示
  - 配方选择
  - 调酒循环音效
- 当前调酒成功判定为：
  - 以 `preferred_drink.formula` 或教学配方为目标
  - 做“精确配方匹配”
- 已支持识别已知配方并解锁酒谱。

#### F. 奖励、库存、酒谱

- 已有 `RewardPhase`。
- 奖励可转化为：
  - 配料
  - 配方
  - 其他可展示奖励
- 已有库存、已解锁配方、奖励新获得提示。

#### G. 日记、日结、图鉴式信息页

- 已有 `Diary`，支持查看当前观察与历史日结。
- 已有 `DaySummaryPhase`，能汇总：
  - 当日来客
  - 调酒结果
  - 线索
  - 奖励
  - 课题
- 已有 `BookModal`，承载：
  - 材料图鉴
  - 酒谱
  - 角色档案/故事入口

#### H. 存档、设置、调试

- 已有 IndexedDB 存档系统。
- 已支持多存档位、读档、删档。
- 已有设置面板与音频设置。
- 已有调试跳转，可直接跳到指定周/天/当日第几位客人。

#### I. 音频系统

- 已有独立音频系统。
- 已支持：
  - 阶段默认音乐
  - 节点音频覆盖
  - BGM/环境音/SFX
  - 循环音效
  - 音量持久化

### 2.2 部分落地、仍有缺口的功能

#### A. 故事解锁机制已开始接入，但消费还不完整

- `story_unlocks.chapters` 已正式进入运行时，并写入 `unlockedStoryChapters`。
- `BookModal` 现以 `gallery.yaml` 为唯一正式档案来源。
- 旧的 `meta.story / character_stories / stories / story_entries` 猜测式读取已不再是正式路径。
- 占位故事策略不再是当前规范的一部分。

结论：故事解锁与图鉴档案的正式路径已经统一，但后续仍需要继续做内容生产层面的补写和整理。

#### B. transcript 已经有了，但还没变成玩法机制

- 当前客人已过剧情可以回看。
- 但 transcript 还只是“记录工具”，没有继续反哺：
  - 剧情回顾
  - 关键词整理
  - LLM 上下文
  - 提示系统

#### C. 调酒结果仍偏硬匹配

- 现在是精确公式匹配。
- `drink_request.keywords`、`hint` 已存在，但更像提示文案，尚未形成“语义层判定”。

结论：当前调酒系统可运行，但若后续想做“情绪调酒”“语义匹配”“更柔性的正确答案”，还需要重构评估层。

### 2.3 当前未正式落地、或资产存在但运行未消费的部分

#### A. NPC 大模型个性化对话已完成 MVP 接线

- 本地后端 `local-backend.mjs` 已提供：
  - `/health`
  - `/api/ping`
  - `POST /api/npc-dialogue`
- 尾声聊天前端已通过正式 `NpcDialogueRequest` 调用本地接口。
- 当前真实模型供应商已切到 MiniMax：
  - `M2-her`
- API key 只存在后端环境变量，不进入前端状态、存档或浏览器存储。

结论：对话能力已经形成“前端结构化请求 -> 本地后端 -> MiniMax -> 结构化响应”的 MVP 闭环。

#### B. 角色扩展文件并未全部进入运行时

当前角色目录中已出现以下扩展文件：

- `nodes_chat.yaml`
- `nodes_hidden.yaml`
- `gallery.yaml`
- `loot_table.yaml`

其中当前已正式接入运行时并受 `content:check` 校验的有：

- `nodes_chat.yaml`
- `nodes_hidden.yaml`
- `gallery.yaml`

尚未正式进入运行时契约的扩展位主要是：

- `loot_table.yaml`

结论：角色扩展位已经不再只是目录占位，但仍有个别扩展文件处于后续待接入状态。

#### C. 图鉴故事页存在占位内容

- 这个问题已在本轮收口中解决。
- 当前图鉴故事页以 `gallery.yaml` 正式章节为准，不再以占位故事掩盖内容缺失。

结论：图鉴档案系统已经完成“正式内容源唯一化”。

## 3. 适合优先补上的剧情推进小功能

以下功能都偏“小而有效”，适合增强叙事推进感，且不会一下子把系统做得太重。

### P0：优先做

#### 1. 来客上次来访回顾

在客人再次来访时，给玩家一个非常短的回顾：

- 上次你观察到了什么
- 上次你给他调了什么
- 上次留下了什么课题

价值：

- 强化连贯感
- 让周目推进更像“持续营业”而不是离散关卡

#### 2. 观察线索 -> 调酒提示 的显式映射

当前观察和调酒之间的联系还偏隐性。建议新增一个轻量提示区：

- 已发现线索
- 可能相关的味道/情绪关键词

价值：

- 降低玩家遗忘成本
- 强化“观察不是摆设”

#### 3. 当前客人 transcript 关键词提炼

现有 transcript 已有基础。可以在其上增加：

- 关键词标签
- 重要句高亮
- 与调酒相关的提示句置顶

价值：

- 减少剧情信息流失
- 为后续 LLM 上下文整理打基础

#### 4. 继续清洗 `nodes_chat.yaml` 的入口语义

`nodes_chat.yaml` 已经正式接入运行时。  
当前更值得继续优化的，不是“有没有接入”，而是“入口是否足够准确”。

建议规则：

- 只在当前主线节点允许侧出时进入
- 不直接改变主线节点流
- 主要承担人设丰富、轻线索投放

价值：

- 低风险增强人物存在感
- 为后续更精细的闲聊 / 隐藏叙事分层打基础

### P1：第二批做

#### 5. 把 `nodes_hidden.yaml` 做成隐藏线索/传闻层

适合作为：

- 特定观察完成后触发
- 特定材料/酒谱解锁后触发
- 连续几次成功接待后触发

价值：

- 提高复访动力
- 增强世界观深度

#### 6. 日结里的“课题推进板”

现在日结已经有内容汇总。可继续增强为：

- 本日推进了哪些角色线
- 哪个角色下次来访最值得关注
- 哪些隐藏条件接近满足

价值：

- 强化玩家的长期目标感

#### 7. 客人对“你记得我”产生反馈

如果玩家已经观察过同一角色、调对过相关酒、解锁过故事片段，则在下次来访时加入一两句反馈。

价值：

- 强化陪伴感
- 让系统性进度转化成情感反馈

## 4. NPC 大模型个性化对话的建议落地方式

### 4.1 不建议直接让模型接管主线

主线剧情仍应由 YAML 节点控制。  
LLM 更适合承担：

- 闲聊
- 补充回应
- 观察后的轻反馈
- 调酒后的个性化短评
- 主线间隙的可选对话

原因：

- 主线可控性强
- 不破坏既有剧情节奏
- 便于回退和调试

### 4.2 建议的第一版接入边界

第一版只做一个安全场景：

- 在 `nodes_chat.yaml` 或“可选闲聊按钮”中，调用 NPC 个性化对话接口

输入上下文建议包含：

- `character_id`
- 角色基础设定摘要
- 当前周/日/来访序号
- 当前剧情阶段/节点
- 已观察线索
- 当前客人的 transcript 摘要
- 最近一次调酒结果
- 明确的输出限制

输出建议固定成 JSON，例如：

```json
{
  "reply": ["……这句话", "……那句话"],
  "mood": "calm",
  "hint": "可选提示",
  "should_unlock_chat_tag": "optional_tag"
}
```

### 4.3 建议的后端结构

不要让前端直接调用模型。建议：

- 前端：只请求本地后端 `/api/npc-dialogue`
- 后端负责：
  - 读取环境变量
  - 拼 prompt
  - 调模型
  - 校验输出
  - 失败时回退到固定文案

当前后端模块边界已经收口为：

- `server/npcDialogue/promptBuilder`
- `server/npcDialogue/schema`
- `server/npcDialogue/provider`
- `server/npcDialogue/route`

`local-backend.mjs` 目前保留为轻量入口，MiniMax 相关逻辑已下沉到目录式模块。

### 4.4 第一版落地原则

必须同时满足：

1. 可关闭：没有 key 时直接报显式错误，不伪造正常回复。
2. 可限制：输出长度、语气、剧情权限都要受控。
3. 可隔离：密钥只存在后端环境变量。
4. 可替换：模型供应商不要和前端或剧情结构硬绑定。
5. 可扩展：后续可再加缓存、few-shot 和更细的人设摘要。

## 5. 目前项目的可迭代性评估

## 5.1 优点

当前项目已经具备不错的可迭代基础：

- 内容与前端逻辑开始分离，方向正确。
- 角色/节点/观察/排班已数据驱动。
- 存档、音频、日结、图鉴等外围系统已经搭好。
- 有调试跳转，对内容验证很有帮助。
- 现有功能闭环已经不止一个 demo 场景。

结论：项目“能迭代”，不是从零散代码起步。

## 5.2 风险

### 风险 1：流程虽然已经机化，但尚未迁移到完整 XState actor 体系

当前项目已经完成了“显式状态图 + reducer + 控制器”的状态机化收口，但还没有迁移到完整 XState actor 体系。

问题：

- 当前核心流程已经清楚，但后续如果要继续引入更复杂的异步子流程，仍可能遇到 actor 边界不够明确的问题。

### 风险 2：内容资产与运行消费不完全对齐

目录里已经有很多扩展位，但运行时目前没有全部消费。

问题：

- 作者会以为“写了就生效”
- 实际运行时却没有接进去

这是最容易让后续内容生产失控的点之一。

### 风险 3：缺少运行时内容校验

现在 YAML 可以被加载，但缺少统一的 schema 校验和错误报告。

问题：

- 字段写错时，不一定第一时间暴露
- 角色一多，排错成本会急剧上升

### 风险 4：图鉴/故事页存在占位回退

这个风险已在本轮被显著降低。

当前问题已从“占位故事掩盖缺失”转成“正式章节内容是否写全”的生产问题，而不是运行时契约问题。

### 风险 5：后端尚未形成真正的能力层

一旦接入 LLM、缓存、日志、安全开关，本地后端现在的结构很快不够用。

## 5.3 要保证“可迭代”，建议立刻坚持的约束

### 约束 A：角色扩展位要么接入，要么明确标注未接入

建议把每种资产文件状态写清楚：

- 已运行消费
- 预留未接入
- 已废弃

不要长期处于“目录里有，但没人知道有没有生效”的状态。

### 约束 B：新增 YAML 字段必须同步补校验

建议新增一个内容校验层，至少校验：

- 必填字段
- 节点跳转目标是否存在
- `schedule.yaml` 引用的 `start_node` 是否存在
- `story_unlocks` 指向的章节 id 是否存在
- 奖励 id / 配方 id / 材料 id 是否可解析

### 约束 C：主线节点与 LLM 对话分层

建议长期坚持：

- 主线：手写/配置驱动
- 模型：只做补充对话和可选互动

这样才能保证剧情始终可控、可回放、可测试。

### 约束 D：尽快拆分主状态管理

建议后续逐步从 `App.tsx` 中拆出：

- `useGameSession`
- `useNarrativeProgression`
- `useGuestFlow`
- `useInventoryAndRewards`

目标不是一次性大重构，而是让后续新增功能有明确落点。

### 约束 E：补一套项目级文档，而不是只有剧情规范

建议至少保留以下文档：

1. `剧情文件规范文档.md`
2. `状态系统驱动内容规范.md`
3. `状态系统驱动节点模板.md`
4. `项目系统与迭代说明.md`
5. `LLM开发需求文档.md`
6. `MiniMax对话模块说明.md`
7. 后续新增：`内容校验与调试清单.md`

## 6. 下一步建议

建议按照下面顺序推进：

1. 先把当前“状态系统已完成到哪一步、哪些仍是非目标”的边界固化。
2. 继续做节点入口语义清洗，例如闲聊入口只在当前节点允许侧出时出现。
3. 再决定是否进入完整 XState actor 迁移，而不是一边改内容一边改底层流程。
4. 大模型对话继续延后，不与当前主流程规范收口混做一轮。
5. 后续再评估 `loot_table.yaml` 等剩余扩展位接入。

## 6.1 本轮收尾结果

本轮迭代已经完成的收尾项有：

- 状态系统主流程、存档快照、内容契约与前置校验已收口。
- `content:check` 已能通过。
- `gallery.yaml` 已成为图鉴故事的唯一正式来源。
- 全局字体已统一回到 `Zpix` 主路径。
- 营业中的壳层背景已修正，不再错误复用开始页背景。
- 右上角“闲聊”按钮已收紧为节点末端、无待处理分支时才允许侧出进入，不再是全局常驻入口。

## 7. 当前结论

这个项目已经具备继续做大的基础，但现在最大的挑战不是“还能不能加功能”，而是“新功能接进来后，会不会让内容系统和运行系统越来越脱节”。

所以当前最重要的方向不是无序加内容，而是：

- 把真实运行边界说清楚
- 把 LLM 接入边界定清楚
- 把扩展位和校验机制补起来

只要这三件事处理好，后续新增角色、隐藏线、闲聊、个性化对话，都会明显更容易做。
