# 《黄昏居酒屋》状态系统驱动内容规范

更新时间：2026-04-18  
文档定位：当前运行时真相、内容契约与状态系统边界说明  
优先级：当本文件与 `剧情文件规范文档.md` 冲突时，以本文件为准

## 1. 目标

这份文档不再回答“理想中的剧情 YAML 应该长什么样”，而是回答：

- 当前项目的状态系统真实如何推进营业流程。
- 内容文件到底在哪些状态里被消费。
- 哪些字段是正式契约，哪些只是历史遗留兼容字段。
- 哪些内容必须通过前置校验，不能再依赖运行时兜底。

从 2026-04-17 起，内容生产和内容评审都应优先遵循这份文档。

## 2. 运行时真相

当前游戏运行时只有一个业务真相：

- `GameSnapshot.value`
- `GameSnapshot.context`

定义位置：

- [src/state/gameState.ts](F:\twilight_izakaya\src\state\gameState.ts)

### 2.1 根状态图

当前正式根状态值固定为：

- `boot`
- `startScreen`
- `mainMenu`
- `introSequence`
- `dayLoop.intro`
- `dayLoop.guest.story`
- `dayLoop.guest.chat`
- `dayLoop.guest.observation`
- `dayLoop.guest.mixing`
- `dayLoop.guest.result`
- `dayLoop.guest.reward`
- `dayLoop.guest.reflection`
- `dayLoop.daySummary`

合法迁移定义位置：

- [src/state/gameTransitions.ts](F:\twilight_izakaya\src\state\gameTransitions.ts)

结论：

- 内容不再直接驱动 `phase + 多个布尔值`。
- 内容只能通过节点字段影响状态机控制器的下一步行为。
- 非法状态跳转属于代码或内容设计错误，不允许靠运行时猜测修复。

### 2.2 哪些阶段由内容驱动

以下阶段直接消费角色内容：

- `dayLoop.guest.story`
- `dayLoop.guest.chat`
- `dayLoop.guest.observation`
- `dayLoop.guest.mixing`

以下阶段主要由运行时生成，不是手写剧情节点：

- `dayLoop.guest.result`
- `dayLoop.guest.reward`
- `dayLoop.guest.reflection`
- `dayLoop.daySummary`

这意味着：

- `result / reward / reflection / daySummary` 不是新的 YAML 节点类型。
- 这些界面由当前节点产物、调酒结果、奖励列表、日记记录和日汇总驱动。

## 4. 文件职责

### `character_meta.yaml`

负责角色静态信息：

- `base_info.name`
- `base_info.type`
- `base_info.description`
- 可选角色偏好，如 `drinks`

不负责：

- 正式故事章节正文
- 运行时占位故事

### `gallery.yaml`

是角色档案和故事章节的唯一正式来源。

正式字段：

- `base_info`
- `profile_sections[].id`
- `profile_sections[].title` 或 `name`
- `profile_sections[].content`
- `profile_sections[].depth_level`

不允许再从以下旧路径猜测故事内容：

- `meta.story`
- `character_stories`
- `stories`
- `story_entries`

### `nodes_main.yaml`

主线剧情节点。  
默认在 `dayLoop.guest.story` 消费。

### `nodes_teaching.yaml`

教学调酒与教学链节点。  
可从 `story` 进入 `mixing`，也可以产出奖励和阶段收束。

### `nodes_chat.yaml`

闲聊节点。  
默认在 `dayLoop.guest.chat` 消费，完成后通常回到原主线节点。

### `nodes_hidden.yaml`

隐藏补充节点。  
运行结构与 `chat` 相同，但语义上承担隐藏线索、补充叙事或特殊观察结果。

### `observations.yaml`

观察点定义。  
只为观察阶段提供特征集合与文案，不直接负责状态推进。

### `schedule.yaml`

营业排班真相来源。  
每个来客入口必须通过 `start_node` 指向一个真实存在的节点。

## 5. 节点契约

正式类型定义：

- [src/data/content/types.ts](F:\twilight_izakaya\src\data\content\types.ts)

### 5.1 所有节点共通字段

推荐所有节点显式写：

- `event_id`
- `trigger_condition`
- `next_node`

当前归一化层允许 `event_id` 或 `id` 至少存在一个，但正式写法仍建议统一使用 `event_id` 作为跨节点引用真名，`id` 仅作为局部语义别名。

### 5.2 `main` / `teaching` 节点

这两类节点必须以 `script_flow` 作为正文承载。

正式要求：

- `script_flow` 必须是非空数组
- 每个步骤 `type` 只能是：
  - `env`
  - `npc`
  - `inner`

运行规则：

- `StoryPhase` 优先读取 `script_flow`
- 不再依赖旧的 `npc_talking / player_inner / bartender_reflection / bartender_inner`

### 5.3 `chat` / `hidden` 节点

这两类节点必须使用环境白描加正文的双层结构：

- `atmosphere_lines`
- `content`

正式要求：

- `atmosphere_lines` 必须为非空字符串数组
- `content` 必须为非空字符串数组

运行规则：

- `StoryPhase` 会把 `atmosphere_lines` 渲染为 `env`
- `content` 会按 NPC 正文渲染

### 5.4 `player_options`

正式字段：

- `text`
- 可选 `option`
- `branch_type` 或 `branchType`
- 可选 `script_flow`
- 可选 `next_node`
- 可选 `fallback_node`
- 可选 `impact_log`
- 可选 `trigger_event`
- 可选 `condition.need_item`

分支语义：

- `flavor`：态度性回应
- `choice`：穷举式探索
- `plot`：显式分支跳转

### 5.5 观察触发

正式字段：

- `trigger_observation.prompt`
- `trigger_observation.continue_node`
- 可选 `trigger_observation.feature_groups`

语义：

- 节点正文讲完后进入 `dayLoop.guest.observation`
- 观察完成后回到 `continue_node`

### 5.6 调酒触发

当前运行时把下列字段视为“混酒节点”信号：

- `drink_request`
- `on_mixing_complete`
- `on_mixing_fail`
- 或 teaching 节点经 `next_node` 指向一段调酒节点

#### `drink_request` 正式要求

- `request_text`
- `preferred_drink.id`
- `preferred_drink.name`
- `preferred_drink.formula`

至少还需要以下其一：

- `on_mixing_complete`
- `on_mixing_fail`

补充规则：

- 若未显式写成功出口，运行时仍可能从依赖该节点的结果节点中按名称推断：
  - `most_loved_success`
  - `generally_liked_success`
  - `regular_success`
- 但这是运行时兼容逻辑，不应作为新增内容的主要写法

#### `teaching` 正式要求

- `teaching.recipe.id`
- `teaching.recipe.name`
- `teaching.recipe.formula`
- `teaching.recipe.steps`

可选：

- `mixing_steps_display`
- `game_action`
- `tip`

### 5.7 奖励与故事解锁

正式奖励字段：

- `reward.details`

支持内容：

- 单个奖励对象
- 奖励对象数组

故事解锁正式字段：

- `story_unlocks.chapters[].id`
- 可选 `story_unlocks.chapters[].reason`

强约束：

- `story_unlocks` 只能指向 `gallery.yaml` 中可解析的正式章节
- 不允许再依赖占位故事页兜底

## 6. 状态系统对内容的真实约束

### 6.1 聊天节点不能接管主流程真相

当前 `chat` 的状态意义是“从主线临时侧出，再回到主线”。

因此默认规则应是：

- 进入 `chat` 前记录 `returnNodeId`
- 聊天结束后返回原故事节点或进入显式指定的后续节点
- 不应把 `chat` 写成一整条替代主线

补充说明：

- 右上角的“闲聊”按钮不再是“只要这个客人有 chat 节点就永久显示”的全局入口。
- 当前运行时只会在 `dayLoop.guest.story` 中显示该按钮。
- 且只有当当前主线节点已经播完最后一句、文本不再打字中、没有待选项、没有待展示奖励、没有待进入观察或调酒时，才允许从这个节点侧出进入闲聊。

这意味着：

- “闲聊入口是否显示”是当前 `StoryPhase` 的运行时可用性判断，不是一个持久化剧情事实。
- 当前规范还没有为主线节点增加新的 `allow_chat` 一类字段；是否可侧出闲聊，暂时由运行时根据当前节点收束状态判断。

### 6.2 奖励、结果、反思由运行时承接

内容作者不需要再写这些旧式节点类型：

- `mixing`
- `reward`
- `transition`
- `ending`
- `system`

当前项目里真正生效的不是“节点 type”，而是“字段触发哪条状态机路径”。

更准确地说：

- 有 `trigger_observation` -> 进入 observation
- 有 `drink_request` / mixing 字段 -> 进入 mixing
- 有 `reward.details` -> 进入 reward 展示
- 有 `diary_note` -> 可能进入 guest reflection

### 6.3 存档保存的是快照，不是页面状态

当前持久化保存的是：

- `state`
- `context`
- `version`

定义位置：

- [src/state/gamePersistence.ts](F:\twilight_izakaya\src\state\gamePersistence.ts)

不进入持久化的内容：

- 日记弹窗是否打开
- 图鉴弹窗是否打开
- 设置弹窗是否打开
- transcript UI 是否展开
- 闲聊按钮当前是否可点击

这意味着内容规范不应该假设“某个弹窗状态”是业务真相的一部分。

## 7. 不再采用的旧原则

以下旧原则已不再适合作为正式规范：

- “字段缺失时尽量降级运行”
- “只要前端能猜出来就先跑起来”
- “节点 type 决定运行阶段”
- “图鉴故事可以用占位内容先补上”

新的约束是：

- 内容错误应尽早暴露
- 契约冲突应由 `content:check` 阻断
- 运行时只消费正式字段

## 8. 校验与提交流程

当前前置命令已经接入：

- `npm run content:check`
- `npm run lint`
- `npm run build`

其中：

- `predev` 会先跑 `content:check`
- `prelint` 会先跑 `content:check`
- `prebuild` 会先跑 `content:check`

结论：

- 新增或修改剧情内容后，必须先过内容校验
- 不能再依赖“运行起来看看哪段炸了”这种后验方式
